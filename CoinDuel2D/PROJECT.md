# CoinDuel2D 项目文档

## 当前定位

`CoinDuel2D` 是一个基于 `Cocos Creator 3.8.8` 的 2D 小游戏项目，当前目标是优先服务移动端玩家，并为后续全平台发布做准备。项目已经全面转向 `Cocos Creator` 小游戏编辑器工作流，用于满足移动端操作、打包和平台适配需求。早期 `Babylon.js`、`Havok` 或网页 P2P 原型只作为玩法参考，不作为当前技术路线。

当前工程处于早期玩法原型阶段，核心集中在硬币物理、桌面边界检测和结算钩子。正式规则、平台打包、商业化、美术方向和发布要求仍需继续补充。

## 核心玩法

产品方向是“三分钟一局”的休闲竞技小游戏。玩家在微缩圆桌箱庭中，通过瞄准、蓄力和弹射硬币进行对抗，依靠力度控制、碰撞判断、环境变化和随机道具取得优势。

游戏由以下部分组合：

- `CoinDuel` 核心弹射规则。
- 动态箱庭环境，包括天气、地面状态和机关设施。
- 大乱斗道具，可被丢入场地并产生带随机性的效果。
- 适合移动端的短局竞技节奏。

核心目标：通过弹射硬币得分，并不断把对手逼入更差的局面，让对手无路可走。

预期 `CoinDuel` 规则循环：

1. 瞄准弹射：选择硬币，反方向拖拽蓄力，松手发射。
2. 碰撞判定：根据本次撞到的硬币数量，以及是否有硬币掉出台面来结算。
3. 连击继续：如果精准撞击 1 枚硬币且没有出界，当前玩家得分并继续行动。
4. 接力位置：连击成功后，刚弹出的硬币离场结算；下一发必须从被撞硬币的位置继续出发。
5. 强制换手：没有撞到、撞到多枚或造成硬币掉落，都会立即把回合交给对手。

判定优先级：

| 现象 | 得分结果 | 回合结果 |
| --- | --- | --- |
| 任意硬币掉下桌子 | 掉落硬币按数量给对手计分 | 立即换手 |
| 没撞到任何硬币 | 双方不得分 | 立即换手 |
| 精准撞击 1 枚硬币 | 当前玩家 +1 | 当前玩家继续 |
| 一次撞击多枚硬币 | 刚弹出的硬币给对手计 1 分 | 立即换手 |

基础结算逻辑：

```text
if 检测到桌面上有任何硬币掉落:
    对手加分(掉落硬币个数)
    触发换手
else:
    switch 主动撞击的硬币数量:
        0:
            触发换手
        1:
            当前玩家得分 +1
            拿走刚弹出的硬币
            强制锁定被撞硬币为下一发起手子弹
            保持当前回合
        2 个或更多:
            刚弹出的硬币归对手计 1 分
            触发换手
```

实现注意点：

- 掉落检测优先于撞击数量，并覆盖其它结算。
- “主动撞击的硬币数量”指本次发射硬币直接撞到的硬币数量。
- 精准撞击 1 枚后，玩家不能自由选择下一枚硬币，被撞硬币会成为强制起手子弹。
- 换手后，控制权交给对手；下一枚起手硬币的选择规则需要由回合系统进一步定义。

策略重点：连击是最高效的得分方式，但接力币会变成位置风险。一旦后续失误，对手会立刻获得回合控制权和得分机会。

当前原型方向：玩家操作 2D 桌面上的硬币对象。硬币通过 2D 物理移动，可以掉出圆形桌面边界；游戏等待所有硬币停止后进入结算。

当前实现中的玩法循环：

1. 等待玩家操作。
2. 发射或移动一枚或多枚硬币。
3. 通过 2D 物理模拟硬币运动。
4. 移除超过桌面半径的硬币。
5. 所有剩余硬币停止后执行结算逻辑。
6. 回到等待状态。

仍需确定的规则细节：

- 移动端触控手感、力度曲线、瞄准辅助和取消操作。
- 三分钟赛制的计时、加时和胜负规则。
- 单人训练、本地对战、在线对战或平台匹配的阶段范围。
- 最终分数目标、奖励、成长和排行榜模型。
- 每局如何开始、结束，以及如何进入下一局。
- 动态环境和大乱斗道具如何影响公平性、可读性和策略深度。

## 开发计划

### 第一阶段：单机天梯模式

第一阶段目标是用最低成本做出能够验证 `CoinDuel` 核心规则的单机天梯模式。玩家会不断进入一个又一个关卡，并且必须保持连胜。中间只要失败一次，本轮挑战就结束。

核心动机：

- 给玩家一个简单、可重复挑战的循环。
- 通过连胜压力制造“再来一局”的驱动力。
- 用炫耀和排行榜作为第一层留存机制。
- 在核心玩法得到验证前，暂时避免联机对战的成本和复杂度。

单轮流程：

1. 玩家开始一轮天梯挑战。
2. 游戏加载关卡，包括桌面布局、硬币配置和可选环境变化。
3. 玩家必须按照基础结算规则完成成功连击。
4. 成功后进入下一关。
5. 任意失败都会结束本轮挑战，并提交最终连胜数或进度到排行。

第一阶段失败条件：

- 任意硬币掉下桌子。
- 发射硬币撞到超过 1 枚硬币。
- 发射硬币没有撞到有效硬币。
- 后续由关卡、环境或道具规则定义的任何非连击成功条件。

第一阶段排行榜价值：

- 主指标：最高连续胜利关卡数。
- 可选副指标：最快通关时间、总分、最高连击数、最少发射次数或最高难度层级。
- 排行榜需要方便玩家比较和分享结果。

第一阶段实现优先级：

- 可靠的发射输入和力度控制。
- 稳定结算掉落、撞击数量、得分、换手、继续回合和接力币锁定。
- 关卡推进和失败流程。
- 基础结算界面，展示本轮连胜结果。
- 先做本地记录，后续再接平台排行榜。
- 高质量即时反馈，包括震动、音效、特效、碰撞火花、硬币运动打磨、掉落表现、连击庆祝和失败冲击。

第一阶段质量目标：

- 单机天梯必须先做到足够好玩，能够留住玩家，再开始联机范围。
- 每个关键动作都需要清晰反馈：瞄准、蓄力、释放、碰撞、得分、连击继续、接力币锁定、掉落、失败和结算总结。
- 移动端震动要强化高价值瞬间，但不能过度打扰。
- 音效要让硬币碰撞、连击成功、危险和失败都能被立刻理解。
- 特效要强调风险、精准和奖励，但不能遮挡物理运动和判定可读性。

进入联机阶段前的门槛：

- 持续打磨单机版本，直到留存、复玩或排行榜/分享行为至少出现正向迹象。
- 只有当独立核心循环已经具备明显吸引力后，才开始打磨联机或多人版本。
- 联机模式是第二个主要开发阶段，不是验证玩法前的必要前提。

长期爽感方向：

- 游戏要建立高张力的风险/回报循环：每一发都像押上了位置、分数和连胜。
- 奖励反馈要在碰撞结算后形成“开奖式”的即时回报感。
- 失败必须清晰、利落，让玩家知道是自己的判断问题，并愿意立刻重试。
- 核心设计避免真实金钱赌博机制；除非未来单独制定法律、平台和合规方案。

## 引擎与配置

- 引擎：`Cocos Creator 3.8.8`
- 语言：`TypeScript`
- 方向：移动端优先的 `Cocos` 小游戏编辑器工作流
- 主场景：`assets/Levels/MainScene.scene`
- `TypeScript` 配置：`tsconfig.json` 继承 `Cocos` 生成配置，当前 `strict` 为 `false`
- 物理配置：`settings/v2/packages/project.json` 中项目重力 `y` 为 `0`
- 已启用模块包括 2D、UI、audio、tween、graphics、mask、rich text、tiled map、Spine 3.8、DragonBones、video、webview 和 2D physics

## 目录结构

- `assets/Levels/`：场景资源，目前包含 `MainScene.scene`
- `assets/Prefabs/`：可复用 `Cocos` 预制体，目前包含 `Coin.prefab`
- `assets/Resources/`：运行时或导入资源，目前包含 `coin.png`
- `assets/Scripts/`：玩法脚本，目前包含 `GameLogic.ts` 和 `TestImpulse.ts`
- `settings/v2/packages/`：`Cocos` 编辑器和项目设置
- `library/`、`temp/`、`profiles/`：生成目录或编辑器状态，不作为源码依据

## 场景结构

`MainScene` 包含一个 `Canvas`，主要子节点如下：

- `MainCamera`：正交相机，清屏颜色为黑色。
- `GameBackground`：带 `Sprite` 的背景节点。
- `Table`：桌面视觉占位节点。
- `CoinGroup`：当前硬币实例的父节点。
- `GameplayManager`：挂载 `GameLogic` 组件。
- `UIManager`：后续 UI 逻辑的占位管理节点。

`Canvas` 尺寸为 `1280 x 720`，通过 `cc.Canvas` 和 `cc.Widget` 与屏幕对齐。

## 玩法组件

### GameLogic

`assets/Scripts/GameLogic.ts` 定义了基础阶段循环：

- `WAITING_PLAYER`：等待玩家操作。
- `ANIMATING`：监听硬币物理运动，直到所有硬币停止。
- `SETTLING`：预留的结果结算阶段。

编辑器暴露属性：

- `coinGroup`：包含当前硬币的父节点。
- `tableRadius`：硬币掉落判定距离，当前为 `400`。
- `speedThreshold`：静止判定速度阈值，当前为 `0.05`。

当前行为：

- 每帧检查所有硬币。
- 销毁距离场景原点超过 `tableRadius` 的硬币。
- 所有硬币速度低于 `speedThreshold` 后调用 `processResult()`。
- 提供 `startSimulation()`，供后续发射或输入系统调用。

### TestImpulse

`assets/Scripts/TestImpulse.ts` 是原型测试组件。它会等待 `0.5` 秒，然后给挂载节点上的 `RigidBody2D` 施加 `(50, 20)` 的线性冲量。这个脚本适合本地测试物理效果，进入正式版本前需要重新评估。

## 预制体与资源

### Coin Prefab

`assets/Prefabs/Coin.prefab` 定义了 `Coin` 节点，包含：

- `UITransform`：`860 x 819`，缩放 `0.1`
- `Sprite`：使用 `assets/Resources/coin.png`
- `RigidBody2D`：动态刚体，`linearDamping` 为 `0.5`，预制体内 `gravityScale` 为 `1`
- `CircleCollider2D`：半径 `370`，`friction` 为 `0.2`，`restitution` 为 `0.8`

场景中的硬币实例覆盖了 `gravityScale` 为 `0`，并放置在约 `x: 14.186` 的位置。

### coin.png

`assets/Resources/coin.png` 被导入为 `sprite-frame`：

- 尺寸：`860 x 819`
- 启用 Alpha
- 过滤：linear
- 包裹模式：clamp-to-edge
- `pixelsToUnit`：`100`
